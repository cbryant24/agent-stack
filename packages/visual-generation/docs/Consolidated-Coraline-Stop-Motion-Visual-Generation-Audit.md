# Consolidated Audit — Coraline Stop-Motion Visual Generation Pipeline

> **Amendment (2026-07-15):** Set continuity is now **plate-first**. This film uses a small number of fixed camera setups, so approved still plates (one per framing, derived from a master plate per set) are the authoritative set canon. The persistent Blender/3D set layer is **demoted from a Phase-2 prerequisite to a conditional escalation**: it is built only as a Test D comparison arm or if an explicit trigger fires (see §9). The 3D skill set is being pursued separately as its own agent/learning project, decoupled from this film's critical path.

## Executive Verdict

**NO-GO on the current production architecture.** The existing system cannot reliably achieve the stated goal of “the same puppet, the same set, every shot” because its source of truth is still probabilistic text generation: long prompt assembly, global character LoRAs, and unconstrained fresh image synthesis. The core failure is an **architecture-level conditioning and asset-authority gap**, not a prompt-writing problem.

The project should **not** be restarted around another image model alone. It should be rebuilt around **persistent canonical assets and constrained image transformation**, with:

1. one approved hero design and coherent multi-view reference package per character;
2. one approved **master plate** per recurring set, plus versioned derived plates for each required framing — persistent 3D geometry is a conditional escalation when plate-based continuity fails a defined gate (§9);
3. explicit pose, camera, mask, and subject-region control;
4. sequential character placement rather than two globally applied identity LoRAs;
5. AI image models used as controlled editors and finishers, not as the authority for identity or camera continuity — set-geometry authority is the approved plate pixels, protected by masks and validated after each edit (Gate 2A).

Z-Image-Turbo should remain available for ideation, rough visual development, and non-canonical exploration. It should be removed from the authoritative continuity path.

# 1. Critical Evidence Correction

The five strongest-looking Celeste pose images included with sidecar `.txt` captions were **training inputs**, not outputs generated by the trained Celeste v2 LoRA.

This matters because two earlier arguments depended on a false contrast:

> “Short prompts plus the same model and LoRA produce strong, consistent solo outputs, therefore the model and LoRA data are not major causes.”

That conclusion is unsupported. Those five images demonstrate what the LoRA was trained on, not what the LoRA reliably reproduces.

The confirmed v2 LoRA outputs in the supplied material are the fixed-seed QKV calibration renders and the shots in the production batches. Those outputs remain noticeably waxy and show identity drift. The calibration portrait also includes rosy cheek coloration despite the intended design forbidding blush.

## Consequences of this correction

The following earlier claims must be removed or downgraded:

- “The v2 solo recipe is already validated.”
- “The solo LoRA is proven production-ready.”
- “The clean solo contrast refutes model or training-data attribution.”
- “Freeze the winning Z-Image solo recipe as a canonical generator.”
- “Increase LoRA rank to approximately 64 as the main next training improvement.”

The stronger conclusion is:

> The current LoRA may encode a recognizable Celeste-like character class, but the supplied evidence does not establish a production-stable puppet identity.

# 2. Stated Production Goal

The system must produce film-usable stills and eventual video frames in which:

- Celeste reads as the same fabricated puppet in every shot;
- the narrator reads as the same fabricated puppet in every shot;
- both identities remain distinct in shared frames;
- the sports bar and living room remain the same physical sets;
- camera position and lens behavior are controllable and repeatable;
- poses, interactions, props, and occlusions are controllable;
- the image reads as photographed stop-motion miniature work rather than smooth CGI;
- shot-to-shot continuity is strong enough for editing into a continuous short film.

This is an **asset-continuity and shot-production problem**, not merely an image-quality problem.

# 3. Current Pipeline

The existing production path can be summarized as:

```text
Narrative intent
→ LLM-generated prose prompt
→ alias detection
→ append character canon text
→ append set canon text
→ append style canon text
→ blindly remove forbidden substrings
→ apply one or two character LoRAs globally
→ submit a text2img / img2img / inpaint graph to ComfyUI
→ save output
→ optionally perform another diffusion pass for correction or integration
```

The surrounding engineering is stronger than the visual-control core:

- Click CLI separation of drafting and generation;
- RunPod and ComfyUI integration;
- workflow registration;
- generation records;
- source and mask tracking;
- cost/provenance logging;
- report and reaction storage;
- asset lineage.

Those parts are reusable.

The central problem is that the system has no authoritative set state, framing asset,
protected background, or character-region assignment — no persistent representation for:

- character identity;
- set state (plate or geometry);
- framing;
- subject-region assignment;
- pose;
- occlusion;
- lighting layout;
- prop placement.

The current “canon” is descriptive text, not a production asset.

# 4. Root Cause

## Primary classification

**Architecture / conditioning-tooling gap.**

## More precise formulation

The pipeline has no authoritative visual-spatial state. It substitutes:

| Required production control | Current substitute | Result |
|---|---|---|
| Exact puppet identity | LoRA + repeated text | Approximate character category |
| Same set geometry | Long set prose | Similar theme, different room |
| Camera repeatability | Prompt wording | Uncontrolled composition |
| Subject-region identity binding | Two global LoRAs | Attribute transfer and identity collapse |
| Actor blocking | Positional prose | Unreliable pose and staging |
| Stable materials | Style words | Waxy or CGI-biased output |
| Continuity across shots | Seed and prompt iteration | Re-roll churn |

The missing capability is not “a better adjective.” It is persistent, versioned visual assets and explicit spatial controls.

# 5. Text Canon Audit

## What text canon can do

Text canon can improve the probability of broad semantic traits:

- dark curly bun;
- black collared shirt;
- button eyes;
- warm sports-bar lighting;
- terracotta walls;
- red-and-white shoes.

## What it cannot do

It cannot preserve:

- exact skull geometry;
- identical button-eye diameter and spacing;
- identical nose and jaw;
- exact hair topology;
- body proportions;
- exact clothing construction;
- exact room dimensions;
- furniture coordinates;
- camera transform;
- character-region assignment.

The existing mechanism is therefore misnamed. It is a **prompt macro layer**, not canon.

## Blind forbid stripping is actively harmful

The substring-removal system produces malformed and contradictory text such as:

- `clean-shaven with no,`
- `NOT, NOT blue-toned`
- incomplete clauses after removing a forbidden word;
- removal of terms from otherwise valid instructions;
- duplicated negations in a positive-only prompt.

This should be deleted, not repaired incrementally.

## Correct replacement

Use structured metadata and validators:

```yaml
character:
  id: celeste_v1
  wardrobe: black_bar_uniform_v1
  hair: bun_front_curl_v1
  eyes: black_four_hole_buttons_v1
  reference_bundle: celeste_reference_pack_v1

set:
  id: sports_bar_v1
  camera: bar_counter_medium_01
  lighting: amber_night_v1

shot:
  pose_reference: pose_05
  character_regions:
    celeste: mask_celeste
    narrator: mask_narrator
```

The source of truth should point to actual assets and controls, not repeat physical descriptions.

# 6. Multi-Character Failure

## Why the two-LoRA path fails

Both character LoRAs are applied globally to shared model activations. The workflow has no hard assignment that says:

- narrator identity belongs only to the left body;
- Celeste identity belongs only to the right body;
- narrator shoes belong only to the narrator;
- Celeste hair belongs only to Celeste.

Prompt phrases such as “left,” “right,” “must not look alike,” and detailed clothing descriptions are weak semantic guidance. They are not spatial routing.

Observed consequences match this structure:

- Celeste inherits narrator footwear;
- both figures converge toward similar faces;
- narrator proportions or hair collapse;
- background figures inherit foreground identity traits;
- two-character frames are materially less stable than solo frames.

## What should be abandoned

- global two-character LoRA chaining as a production solution;
- seed sweeps intended to find a lucky clean two-shot;
- LoRA-strength balancing as a substitute for identity isolation;
- whole-frame harmonization passes that reload both character LoRAs.

## Correct near-term architecture

Use one character per controlled edit pass:

1. start from a locked background or rendered set plate;
2. define blocking with pose/depth/rough geometry;
3. insert narrator into a narrator-only mask using narrator references;
4. preserve narrator and set;
5. insert Celeste into a separate mask using Celeste references;
6. correct overlaps, shadows, hands, props, and eye details locally;
7. perform only a low-change finishing pass, if needed.

Regional conditioning can be evaluated later, but sequential masked placement is easier to validate because each pass has one identity and one region.

**Explicit fallback if masked insertion underperforms:** generate or edit each character against
a controlled background, extract an alpha/matte, composite conventionally into the plate, and
use diffusion only for local grounding and shadow integration. This can be more stable than
asking one edit model to simultaneously preserve the plate, pose the character, hold identity,
and invent correct shadows.

# 7. Character Identity and Training-Data Audit

## Current dataset problem

The Celeste LoRA was trained on approximately eleven synthetic images, including AI-generated and image-converted frames. The supplied training examples already vary in:

- face width;
- jaw and chin shape;
- ear size;
- button-eye construction;
- eyebrow shape;
- nose;
- bun size;
- front-curl placement;
- hairline;
- body proportion;
- outfit detail;
- jewelry;
- rendering style;
- set and lighting.

A LoRA trained on this material does not learn one exact puppet. It learns a distribution resembling:

> dark-haired button-eyed woman in black clothing, often in a sports-bar setting.

That is a character class, not a locked production asset.

## Important nuance

The problem is not simply that there are only eleven images.

A coherent, controlled set of 10–15 images can sometimes train a useful identity. A larger inconsistent set can perform worse. The principal deficiencies are:

- no single approved base geometry;
- no true coherent multi-view package;
- no controlled front/profile/rear views;
- no consistent expression set;
- no background-neutral coverage;
- no clean separation of identity, wardrobe, and setting;
- no held-out validation images.

## Rank is not the main issue

Increasing LoRA rank is not the next correct move.

Higher rank can:

- increase memorization of inconsistent anatomy;
- strengthen background correlations;
- increase training cost;
- overfit synthetic rendering artifacts;
- fail to add missing viewpoints or geometry.

Do not retrain the current dataset at rank 64. Fix the source identity package first.

## Required character source of truth

Each character needs one approved hero design and a coherent reference bundle:

- neutral front portrait;
- left and right three-quarter portraits;
- left and right profiles;
- rear head/hair view;
- full-body front;
- full-body side;
- full-body rear;
- neutral expression;
- smile;
- smirk;
- laugh/open mouth;
- worried expression;
- consistent lighting;
- consistent materials;
- identical wardrobe and shoes.

All references must depict the same underlying puppet design.

Reference conditioning becomes the primary identity mechanism. A new LoRA may later serve as a secondary prior after the reference package is coherent and validated.

# 8. Model Fit

## Z-Image-Turbo

Z-Image-Turbo remains useful for:

- ideation;
- thumbnails;
- style exploration;
- rough solo concepts;
- storyboard frames;
- non-continuity inserts;
- candidate design generation.

It is not the correct authoritative base for final recurring shots under the current configuration:

- distilled low-step generation;
- guidance effectively disabled;
- no conventional negative prompt;
- no current regional identity routing;
- no current pose/depth/camera controls in the active workflows;
- no reference-conditioned identity path in the active workflows.

## Plastic / CGI look

The waxy result cannot be attributed solely to prompt bloat, because a compact calibration prompt still produced a polished CGI-like portrait.

It also cannot be attributed solely to Z-Image-Turbo. The likely causes are combined:

- base-model prior;
- synthetic LoRA data;
- smooth clay-resin and satin-sheen language;
- studio-like lighting;
- weak miniature-photography cues;
- limited material diversity in training references.

The correct conclusion is:

> The plastic look survives prompt cleanup, so it is not only a prompt problem. It must be addressed through reference design, material assets, lighting, and possibly a different edit/finishing model.

## Qwen-Image-Edit

Qwen-Image-Edit-2511 is a strong candidate for:

- deriving a canonical puppet from real-person references;
- generating coherent multi-view variants from one approved hero;
- inserting one character into a locked plate;
- controlled expression changes;
- wardrobe-preserving edits;
- local material and lighting corrections;
- multi-image fusion experiments.

It is not yet proven as the complete production solution.

## FLUX Kontext

FLUX Kontext should be benchmarked as an alternative editing and finishing model for:

- source-image preservation;
- successive edit stability;
- local material changes;
- expression edits;
- geometry preservation;
- reference-based object and character editing.

## Control-guided generation

A depth/edge/pose-controlled workflow should be evaluated for preserving a rendered shot layout. This may use Qwen ControlNet, another suitable current control model, or a model-specific equivalent.

The correct comparison is not:

> Z-Image versus Qwen versus Flux from blank text.

It is:

> Which editing/control stack best preserves a known render, known character references, and known geometry while adding the desired stop-motion finish?

# 9. Set and Camera Continuity

## Fixed plates: useful but limited

An immutable set plate is useful for:

- repeated shots from the same angle;
- dialogue coverage with a fixed camera;
- local character insertion;
- maintaining the same bar arrangement;
- avoiding fresh text-to-image reconstruction.

A single plate does not solve:

- arbitrary camera movement;
- new lens angles;
- threshold-to-interior transitions;
- wide-to-close spatial continuity;
- video camera motion.

**This film requires none of those.** The production design is a small number of fixed camera setups with no camera movement; the video model animates characters, not the camera. Plates are therefore the primary set canon for this production:

- one **master plate** per recurring set, generated empty and director-approved;
- additional framings **derived from the master plate** with the edit model ("same room, viewed from…") and individually approved, so all framings describe one physical room (Test B2);
- depth and segmentation for control extracted **from the plate itself** (monocular depth + SAM-class masks) — a **soft structural guide**, not authoritative geometry: estimated relative depth can err around glass, mirrors, TV screens, thin stool legs, and blurred background objects, and its value must be validated experimentally (this is also the clearest plate-vs-Scene-Forge distinction: plate depth is inexpensive and approximate, Blender depth is deterministic and geometrically coherent);
- every keyframe in a scene produced by editing that scene's plate — reusing the same plate makes within-framing set drift **preventable and measurable**, not automatic: an edit model can still alter bottles, furniture edges, TV shapes, fixtures, and wall texture, so background regions must be protected through inverse masks or layered compositing and validated after each edit (Gate 2A).

## Conditional escalation: persistent 3D set

Do **not** build this up front. Escalate to a persistent geometric set only if one of these triggers fires:

1. Test B2 cannot derive coherent secondary framings from a master plate (the framings won't read as one room);
2. Test C repeatedly fails shadow grounding / occlusion after local correction passes;
3. the film's design grows real camera moves or shot-reverse-shot coverage;
4. the video phase shows set shimmer that plate-pinned keyframes cannot hold.

If a trigger fires, create a persistent geometric set in Blender or an equivalent scene system.

The model can begin as a simplified blockout. It must preserve:

- floor plan;
- walls;
- windows and doors;
- bar counter;
- furniture;
- televisions;
- lights;
- character scale;
- camera transforms.

For each shot, store:

```text
set asset version
camera transform
lens and sensor
character transforms
pose references
prop locations
lighting state
render settings
```

This is actual set canon.

## Why a 3D source matters

It provides deterministic:

- camera repeatability;
- perspective;
- occlusion;
- depth;
- furniture placement;
- character scale;
- lighting direction;
- shadow grounding;
- animation continuity.

The 3D render does not need to be the final aesthetic. It can provide:

- beauty pass;
- depth;
- normals;
- object masks;
- character masks;
- shadows;
- diffuse/albedo;
- motion vectors.

AI editing then finishes a controlled image rather than reconstructing the set.

# 10. Recommended Production Architecture

## Core principle

**Persistent assets first; AI finishing second.**

## Character layer

- approved hero puppet;
- coherent multi-view reference bundle;
- explicit wardrobe versions;
- expression references;
- character-specific masks;
- no globally chained identity LoRAs.

## Set layer

- approved still **plates** as the authoritative set canon: one master plate per set, plus approved derived plates per framing (~8 total for this film);
- depth and segmentation passes extracted from each approved plate;
- stable prop placement frozen in the plate pixels;
- persistent Blender set / camera library / lighting presets **only on trigger** (§9 conditional escalation), or as a Test D comparison arm.

## Shot layer

Shots are defined in a small **neutral schema shared with Scene-Forge** (shot_id, set_asset,
framing_asset, characters, pose_guides, regions, props, lighting_intent, outputs). The plate
pipeline interprets `set_asset` as a master plate and `framing_asset` as an approved framing
plate; Scene-Forge interprets them as a .blend and a named Blender camera. Shared vocabulary,
independent implementations — an integration seam, not a dependency.

Each shot should be defined by structured data:

```yaml
shot_id: bar_05_toast

set_asset: sports_bar_v1
framing_asset: bar_counter_two_shot_01

characters:
  - character_id: narrator_v1
    reference_pack: narrator_refs_v1
    pose_guide: narrator_toast_v1
    region: narrator_mask

  - character_id: celeste_v1
    reference_pack: celeste_refs_v1
    pose_guide: celeste_laugh_v1
    region: celeste_mask

props:
  - tequila_glass_left
  - tequila_glass_right
  - lime_wedge
  - salt_dish

lighting_intent: amber_night_v1

outputs:
  - final_frame
  - character_masks
  - validation_report
```

This is the canonical form of the shared contract; see `shared-shot-schema.md` for the full
field, versioning, and mapping rules. Neither implementation imports the other — both consume
the schema.

## Rendering and edit layer

Two explicit modes; the film runs in plate production mode unless a §9 trigger fires.

**Plate production mode (default):**

1. load the approved empty framing plate;
2. load or estimate plate depth and segmentation;
3. establish character masks and pose guides;
4. insert the first character;
5. freeze protected regions;
6. insert the second character;
7. correct contacts, shadows, occlusion, hands, eyes, and props locally;
8. composite screens and deterministic graphics conventionally;
9. apply grain, lens treatment, and color finishing;
10. run identity and protected-region validation (Gate 2A).

**Scene-Forge escalation mode (on trigger only):**

1. render beauty, depth, normals, masks, and shadows from the persistent set;
2. use those passes as structural conditioning;
3. apply character-specific edits and material finishing;
4. validate geometry against the source render.

# 13. Required Comparative Tests

Do not commit to Qwen, FLUX, or a control model based on model-card claims alone.

## Test A — hero identity preservation

Input:

- one approved Celeste hero;
- one approved narrator hero.

Tasks:

- front to three-quarter;
- front to profile;
- front to rear;
- neutral to smile;
- neutral to laugh;
- full-body pose change.

Score:

- skull geometry;
- button size and spacing;
- nose and jaw;
- hair topology;
- body proportions;
- wardrobe;
- shoes;
- material consistency.

## Test B1 — successive character-edit drift

Apply three sequential edits to the same character image:

1. change expression;
2. move one arm;
3. alter lighting slightly.

Reject any system that progressively changes identity, hair, wardrobe, set geometry, or camera.

## Test B2 — derived set-framing coherence

Starting from the approved master plate, derive: an alternate wide; a reverse or side angle; a
close framing; a doorway framing. (Run only for sets that actually need multiple framings —
deriving unseen geometry is a harder ask than editing a visible object, and a secondary framing
counts as "derived from the master" only when shared landmarks demonstrably survive.)

Score: room topology; wall/counter relationships; door/window placement; material palette;
recognizable shared landmarks.

## Test C — two-character composition

Use:

- one locked bar plate;
- one narrator reference pack;
- one Celeste reference pack;
- explicit masks and rough pose guide.

Produce:

- narrator solo;
- Celeste solo;
- two-shot;
- interacting two-shot with overlapping arms or props.

Score:

- identity separation;
- shoe ownership;
- hair ownership;
- body proportions;
- face preservation;
- region leakage;
- occlusion;
- shadow grounding.

## Test D — set geometry preservation

Use the approved bar master plate (primary source) — and optionally one gray-box Blender render as a comparison arm — and edit it with:

- Qwen-Image-Edit;
- FLUX Kontext;
- a depth/edge-controlled workflow (depth extracted from the plate; from the blockout only if that arm is built).

Reject any output that materially changes:

- counter position;
- door position;
- furniture coordinates;
- television positions;
- wall geometry;
- camera perspective.

## Test E — material finish

Compare:

1. approved plate baseline;
2. Qwen-edited plate;
3. FLUX-Kontext-edited plate;
4. plate-depth/edge-controlled result;
5. optional Scene-Forge render + finish, when available.

Evaluate:

- handmade irregularity;
- photographed-miniature feel;
- fabric texture;
- clay/resin texture;
- reduced waxiness;
- geometry preservation.

Geometry preservation is the gate. A more beautiful image that changes the puppet or set loses.

# 14. Phased Path Forward

## Phase 0 — stop structurally doomed spend

Immediately stop:

- dual-LoRA generations;
- seed sweeps;
- current-dataset LoRA retraining;
- full eight-shot regeneration;
- pod uptime during review;
- long prompt iteration.

Set the RunPod workflow to stop automatically after queued work drains.

## Phase 1 — establish canonical character assets

For Celeste and narrator:

1. select or create one hero image and take it through the **hero approval gate** — face
   structure, button size/placement, hair topology, proportions, material treatment,
   wardrobe/shoes each explicitly approved; neutral lighting; no background dependence; adequate
   resolution; director sign-off recorded. No derivation starts on an unapproved hero (a flawed
   hero becomes the foundation of every later reference);
2. derive coherent multi-view references from that approved hero;
3. manually reject or correct drift;
4. create expression references;
5. create front/profile/rear full-body references;
6. store them as versioned character reference packs.

Do not train a new LoRA yet.

## Phase 2 — establish canonical set plates

For each recurring set (sports bar, living room, jazz club, storefront):

1. generate the set **empty** at the scene's fixed framing; iterate until director-approved — this is the master plate;
2. derive any additional approved framings from the master plate with the edit model,
   rejecting derivations that change geometry (Test B2). Classify every framing change first:
   **crop/reframe** (no new geometry), **small viewpoint shift** (limited inferred geometry),
   **large alternate angle** (substantial invented geometry — elevated continuity risk), and
   **reverse angle** (effectively a new plate that must be reconciled through landmarks and
   layout rules — the source image contains no information about unseen walls, so this is
   controlled invention, not derivation). The classification determines when Test B2 suffices
   and when the §9 Blender trigger fires;
3. extract depth (monocular) and segmentation (SAM-class) passes from each approved plate;
4. version the plates + passes as the set's canonical reference pack.

No 3D modeling in this phase. A gray-box Blender blockout may be built later as one Test D arm, or on a §9 escalation trigger.

## Phase 3A — editor/control bake-off (reduced tests)

Run reduced-scope Tests A/B1/B2/D across Qwen-Image-Edit-2511, FLUX Kontext, and one
plate-depth/edge-controlled workflow. Select the candidate stack on measured preservation. (The
bake-off determines which stack runs the proof — it cannot come after it.)

## Phase 3B — three-frame proof gate (selected candidate)

Using the same set plate and framing, create:

1. narrator solo;
2. Celeste solo;
3. narrator and Celeste together.

Use sequential masked edits.

Pass requirements:

- both characters read as the same approved puppets;
- neither inherits the other’s hair, shoes, face, or body;
- bar geometry does not change;
- camera does not change;
- material reads as stop-motion rather than polished CGI;
- outputs survive side-by-side review.

If this test fails, do not integrate it into the agent.

## Phase 4 — repeatability validation

Repeat the proof's two-shot with alternate poses and expressions (≥3 variants) on the selected
stack. Gate 3's five-repetition isolation check runs here. A Blender-derived control arm may be
added only if Scene-Forge has reached CP3 or a §9 escalation trigger has fired.

## Phase 5 — agent integration

Only after a successful proof:

- replace prose canon with asset references;
- add structured shot schemas;
- support image references, masks, depth, and segmentation as first-class inputs;
- enforce one-character-per-edit-pass by default;
- add geometry and identity validation;
- add automatic side-by-side contact sheets;
- record editor/model/version for each pass;
- retain exact source lineage.

## Phase 6 — eight-shot still sequence

Rebuild the eight shots using:

- versioned framing plates and their associated framing metadata (shot size, estimated focal character, horizon/perspective notes — no physical camera object is implied);
- approved character references;
- controlled pose guides;
- sequential masked edits;
- conventional compositing for screens and logos.

Do not move to video until the eight stills pass continuity review.

## Phase 7 — video evaluation

Wan 2.2 first/last-frame workflows may be tested only after still continuity succeeds.

Evaluate:

- identity through intermediate frames;
- button-eye stability;
- hair and clothing morphing;
- set geometry;
- body mechanics;
- two-character interaction;
- occlusion;
- temporal flicker.

First/last-frame conditioning constrains endpoints. It does not guarantee the middle.

If FLF2V cannot hold identity/geometry through the middle frames, rig-driven 3D animation is a possible post-capstone escalation — explicitly outside the current film path and the Scene-Forge MVP (see §17).

# 15. Go / No-Go Gates

## Gate 1 — character pack

Go only if one character remains recognizably identical across:

- front;
- profile;
- rear;
- full-body;
- at least three expressions.

## Gate 2 — set preservation

Go only if editing preserves:

- camera;
- wall and counter geometry;
- furniture;
- televisions;
- door/window positions.

## Gate 2A — protected-region preservation (mechanical)

For every character insertion, evaluate three zones — raw global pixel difference is NOT the
metric (grain, color balance, and tiny exposure shifts would fail everything):

1. **Protected background** (outside the dilated character mask): should remain almost
   unchanged — score with structural similarity / perceptual distance, edge displacement around
   fixed landmarks, and explicit object/landmark checks; produce a visual diff image.
2. **Integration band** (narrow dilation around the character mask): lighting and shadows may
   change — manual approval, not a numeric gate.
3. **Editable region** (character + intentional interaction area): unconstrained.

Calibrate the protected-zone thresholds empirically during the three-frame proof, before it is
scored; never rely on one global numeric threshold.

## Gate 3 — two-character isolation

Go only if five repeated two-shot tests show:

- no shoe swap;
- no hair transfer;
- no face convergence;
- no body-type collapse;
- no background identity bleed.

## Gate 4 — chained edits

Go only if three successive edits preserve identity and set geometry.

## Gate 5 — still sequence

Go to video only if all eight stills pass a single contact-sheet review.

# 16. Cost Controls

## Pod discipline

- queue work locally before starting the pod;
- start only for generation;
- automatically stop after the queue drains;
- never leave the pod running during human review;
- perform prompt, mask, and comparison work locally;
- batch tests with fixed matrices;
- define pass/fail criteria before generation.

## Experiment discipline

Every paid experiment must answer one question:

- Does the model preserve Celeste’s face?
- Does the model preserve the bar geometry?
- Does sequential insertion prevent shoe transfer?
- Does a depth map preserve the camera?
- Does the finish improve texture without changing identity?

Do not run open-ended “try another seed” sessions.

## Training discipline

No new LoRA training until:

- a coherent reference pack exists;
- the role of the LoRA is clearly defined;
- validation views are held out;
- rank and step comparisons use identical data;
- the LoRA is not being asked to solve spatial binding.

# 17. Final Architecture Decision

## Film production authority

Use:

- versioned hero and multi-view character reference packs;
- approved master and framing-specific set plates;
- protected-background masks (Gate 2A validation after every insertion);
- plate-derived depth/segmentation as soft structural guides where useful;
- structured pose and blocking references;
- sequential identity-specific edits;
- conventional compositing for deterministic details (screens, logos);
- explicit 3D escalation triggers (§9).

## Optional Scene-Forge authority (on trigger, or if Test D proves it superior)

- persistent set geometry;
- physical cameras and lighting;
- rendered depth, normal, shadow, and mask passes.

## Possible post-capstone extension — not part of the current film path or Scene-Forge MVP

Rigged 3D character proxies or full character assets, and rig-driven animation as a video
backbone. These remain future escalations only; character modeling, rigging, and animation are
explicit non-goals of both the film path and the Scene-Forge MVP.

## AI role

Use AI for:

- concept development;
- puppet design derivation;
- controlled restyling;
- local expression and material changes;
- selective finishing;
- cleanup;
- possibly keyframe-to-video assistance after still validation.

## Z-Image role

Use Z-Image-Turbo for:

- ideation;
- rough shots;
- style exploration;
- storyboard generation;
- non-canonical variants.

Do not use it as the source of truth for recurring production assets.

---

# Final Recommendation

**Rebuild the project around persistent assets, not prompts.**

The next implementation should not be another generalized “generate from prose” agent. It should be a shot-production system that loads:

- a versioned set;
- a versioned camera;
- approved character reference packs;
- pose/blocking data;
- masks and structural controls;
- a selected editor/finisher.

The immediate next step is an approved sports-bar master plate plus hero character references, then the controlled three-frame proof comparing Qwen-Image-Edit, FLUX Kontext, and one depth/edge-guided workflow (depth from the plate). A Blender blockout is built only as an optional Test D comparison arm or on a §9 escalation trigger. No further LoRA training, dual-LoRA rendering, full-batch generation, or video work should occur until that proof passes.
