# Wan 2.2 Still-Count Plan + Image-Pipeline Fact-Check

## TL;DR
- **Use the FLF2V (first-last-frame) approach, not single-still I2V.** For a 4-scene, 4–6 minute short with one static camera per scene, plan for **~52–76 generated keyframe stills** (roughly one per 5-second clip boundary), driving **~48–72 Wan 2.2 clips**. Single-still I2V chaining is simpler but breaks down past ~1 minute of continuous shot due to color/identity drift — disqualifying for your 60–90s-per-scene structure.
- **Generate those keyframes with Qwen-Image-Edit 2511, not Z-Image Turbo + LoRA or FLUX.2.** The central task — take an approved frame and produce an end-frame that is the same camera/character with a changed pose/action — is exactly what an instruction-edit model does. Z-Image Turbo (pure text-to-image) cannot edit a prior frame at all; FLUX.2 is a generation/multi-reference model, less surgical for "same shot, new pose."
- **The document is mostly accurate, with one clear error:** the "reference strength ~0.7 dial (0.6–0.8 workable)" for FLUX.2 dev is NOT a real parameter in local ComfyUI FLUX.2 workflows. It is an artifact of hosted-API/img2img platforms. Local FLUX.2 dev is guidance-distilled and conditions references via chained ReferenceLatent nodes with no per-reference strength dial; the tunable knob is FluxGuidance (guidance_scale, ~4.0 default at ~50 steps), which controls prompt adherence, not reference weight.

## Key Findings

### Part 1 — Wan 2.2
1. **Native clip length is ~5 seconds = 81 frames at 16fps.** Per the official Wan2.2 documentation and community synthesis, 81 frames (~5s) is the 14B "sweet spot" at 16fps where temporal coherence is maintained; users report reliable results up to ~145–161 frames (~9–10s at 16fps) before minor artifacts appear, and pushing toward ~250 frames causes the model to repeat actions or hallucinate. Output is commonly interpolated to 32fps with FILM VFI. Resolutions: 480p (480×832 / 832×480) and 720p. The 5B TI2V variant generates a 5-second 720P video (1280×704 or 704×1280) in under 9 minutes on a single consumer GPU with ≥24GB VRAM (e.g. RTX 4090) per the official Wan2.2 GitHub repo; the 14B I2V models run 480p/720p at 16fps.
2. **Wan 2.2 supports FLF2V natively in ComfyUI** (14B template; 5B via the `Wan22FirstLastFrameToVideoLatent` custom node). FLF2V enforces boundary conditions at t=0 and t=1 inside the diffusion process, so both the start and end stills are respected as ground truths. Per the official ComfyUI FLF2V workflow page, "compared with single-frame I2V, it reduces identity drift and guarantees the final frame matches your target image."
3. **Chaining single-still I2V clips (extract last frame → seed next clip) drifts.** This is the documented, widely-hit failure. GitHub issue kijai/ComfyUI-WanVideoWrapper #1541 reports the generation "tends to increase color contrast over time" and "start[s] brighter than the input image," unfixed by scheduler, VAE, or color-match changes; Wan-Video/Wan2.2 issue #172 reports last-frame chaining where "color often shifted a little bit and it didn't match with original color although I used Color Match." Other modes: saturation buildup, character identity drift, and detail softening from repeated VAE encode/decode. The practical wall is ~1 minute (~12 chained clips) before color shift dominates (MeiGen-AI InfiniteTalk README: "good results from a single image for up to 1 minute. Beyond 1 minute, color shifts become more pronounced").
4. **FLF2V chaining drifts less** because each clip's endpoint is pinned to an approved still (a ground-truth boundary), not improvised. The recommended long-form discipline: pre-generate all approved keyframes first, then FLF2V-interpolate each consecutive pair, ensuring "every video begins where the previous one ends."

### Part 1 — Still-count math (4 scenes, 240–360s, one camera per scene)
- **Clip budget:** 5s clips → **12–18 clips per scene → ~48–72 clips total** (matches your ~50–70 estimate). At 60s/scene → 12 clips/scene = 48 clips; at 90s/scene → 18 clips/scene = 72 clips.
- **Single-still I2V, naive chaining:** 4 scene-anchor stills + ~44–68 extracted last-frames. **Not viable** — each 60–90s scene exceeds the ~1-minute drift wall.
- **Single-still I2V, pragmatic re-anchoring** (regenerate a fresh consistent still every ~2–3 clips instead of propagating drifted frames): **~16–30 generated stills** + extracted frames for intermediate clips.
- **FLF2V keyframe interpolation (recommended):** each clip bounded by a start + end still; consecutive clips within a scene share the boundary frame. A scene of N clips needs N+1 stills. Across 4 scenes (scene cuts don't share frames): **total clips + 4 ≈ 52–76 generated keyframe stills.** (48 clips → 52 stills; 72 clips → 76 stills.)

### Part 2 — Document verification
1. **Qwen-Image-Edit 2511 claims — TRUE.** Enhanced successor to 2509, built on the Qwen-Image 20B MMDiT architecture (per ComfyUI's official blog and RunComfy; released Nov/Dec 2025, Apache 2.0). Native multi-image reference conditioning (character in one image, outfit in another) via the built-in `TextEncodeQwenImageEditPlus` node, up to 3 reference images. Improved character consistency and multi-person fusion (two person references → coherent two-shot) confirmed by the Qwen/HF model card and ComfyUI blog. Native ComfyUI workflow exists. "Integrated LoRA support" = popular community LoRAs baked into the base model — AND stacking your own style LoRA on top is supported (the `Qwen-Image-Edit-2511-Lightning-4steps-V1.0` LoRA itself is loaded via a standard LoRA-loader node). Speed: base 2511 runs ~40 steps and is slower than an 8-step turbo t2i, but the Lightning 4-step LoRA (lightx2v, ~10× speedup vs 40-step) closes the gap. VRAM: full precision ~40GB, FP8 ~20GB, GGUF ~12GB.
2. **FLUX.2 dev claims — TRUE except the strength dial.** 32B-parameter flow-matching transformer with a Mistral-Small-3.2-24B text encoder, released 25 Nov 2025 under a Non-Commercial license (per BFL's official GitHub). Up to 10 reference images (8 via API). VRAM ~64GB BF16 / ~32GB FP8 / ~19GB GGUF Q4_K_S on a 4090 (per Spheron citing city96; note FP8 and Q8 do NOT fit in 24GB — use GGUF Q4 on a 4090). Klein 9B (~29GB FP16 / ~15GB FP8, non-commercial) and Klein 4B (~13GB, Apache 2.0). **The "reference strength ~0.7 dial (0.6–0.8)" is WRONG for local ComfyUI FLUX.2 dev** — references are fused via chained ReferenceLatent nodes with no per-reference strength value; the model is guidance-distilled and the real knob is FluxGuidance (guidance_scale, ~4.0 default). The 0.6–0.8 "strength" figure comes from hosted-API/img2img platforms (where an image-to-image "strength" slider exists), not the local multi-reference graph.
3. **Bottom line — CONSISTENT with mid-2026 consensus:** Qwen-Image-Edit 2511 for per-outfit consistency, two-shots, and shot-to-shot continuity; FLUX.2 for prompt/staging fidelity and photoreal detail. Vendor comparisons echo this ("choose 2511 for pose edits and constrained appearance changes; Flux systems emphasize expressive stylization").
4. **Connection — best model for FLF2V keyframe pairs is Qwen-Image-Edit 2511.** Editing an approved frame into the next beat's end-frame (same camera/character, new pose/action) is a structure-preserving instruction edit — Qwen's core strength. Z-Image Turbo + LoRA **cannot** do this (no image-conditioning input; it is pure text-to-image). FLUX.2 can approximate via reference conditioning but is oriented to fresh generation, not surgical "keep everything, change the pose" edits. Practitioners already use exactly this pattern: generate consistent keyframes with Qwen-Image-Edit, then feed them as first/last frames to a video model for interpolation.

## Details

### Wan 2.2 constraints and motion behavior
The de-facto standard is 81 frames / 5s at 16fps. Community reports (Apatero) attribute Wan's slow-motion tendency to its training on smooth cinematic footage — treat this as a practitioner observation, not an official claim; regardless, the fixes are the same: specify action speed in prompts, generate at 16–24fps rather than 6–8fps preview rates, and tune motion settings. The MoE architecture (high-noise + low-noise experts) is why quality is good per-clip but does nothing to prevent cross-clip drift when chaining.

### Stop-motion / Coraline (Laika) aesthetic interaction
Wan's temporal-consistency smoothing works against the choppy, held-frame look of stop-motion and pulls toward realistic motion and texture. Expect texture shimmer and gradual style drift toward realism across chained clips — the same drift that plagues identity and color. Mitigations: hold frames / target a lower effective frame rate, apply a Laika/stop-motion style LoRA to re-assert the look, keep clips short to bound drift, and lean on FLF2V so the aesthetic is re-pinned at every boundary by an approved, on-style still.

### Character consistency across ~50–70 clips (two characters)
Build a locked reference sheet for each character. Use Qwen-Image-Edit 2511's multi-image reference to compose two-shots from the two locked character references (its multi-person fusion is the 2511 headline improvement). Re-anchor every keyframe from approved stills rather than propagating drifted extracted frames — a practitioner rule of thumb is to refresh the reference roughly every clip (~5s) to keep identity locked. Keep seed, size, length, and prompt tags consistent across clips. Per-character LoRAs remain useful stacked on Qwen. Note: even 2511 retains residual "pixel drift," so visually QA each keyframe before it becomes an FLF2V boundary.

### FLUX.2 local reference workflow (for accuracy of the document)
LoadImage → image scale → VAE encode → chained ReferenceLatent nodes inject each reference into the conditioning stream; you bypass individual ReferenceLatent nodes to A/B test each reference's contribution. Reference conditioning is internal to the model (no IP-Adapter needed). Klein multi-reference reliability can be improved by stitching references into a 2×2 grid before conditioning. There is no per-reference "strength" scalar in this path.

## Recommendations
1. **Adopt FLF2V as the primary pipeline.** Storyboard each 60–90s scene into ~5s beats, pre-generate an approved on-style keyframe for every beat boundary, then FLF2V-interpolate each consecutive pair. Budget **~52–76 keyframe stills** for the full 4–6 minute short.
2. **Migrate keyframe generation to Qwen-Image-Edit 2511.** For each scene, generate the opening keyframe, then produce every subsequent end-frame by editing the prior approved frame with an instruction like "same shot, same characters, [character] now reaching for the door." Stack the `Qwen-Image-Edit-2511-Lightning-4steps` LoRA for speed and your own Laika style LoRA for aesthetic. Constrain edits explicitly ("keep face, keep background, keep camera") to minimize drift.
3. **Keep Z-Image Turbo for ideation only.** It has no image-conditioning input, so it cannot generate FLF2V end-frames from a prior frame. Retire it from the consistency-critical path.
4. **Use FLUX.2 dev only if you specifically need its prompt/staging fidelity** for a few establishing shots and you have the VRAM (FP8 ~32GB on an H100/A100; GGUF Q4 ~19GB on a 4090). It is not required for this project.
5. **Correct the document:** delete the "reference strength ~0.7 (0.6–0.8)" claim for local FLUX.2 dev; replace with "FLUX.2 dev is guidance-distilled; the adherence knob is FluxGuidance (~4.0 default), and references are conditioned via chained ReferenceLatent nodes with no per-reference strength." Keep the "0.6–0.8 strength" note only if the doc is describing a hosted API/img2img endpoint, clearly labeled as such.

### Benchmarks / thresholds that would change the recommendation
- **If a single continuous shot must exceed ~1 minute with zero re-anchoring:** neither I2V nor FLF2V holds cleanly — split the shot or accept periodic re-anchor stills.
- **If Qwen 2511 keyframe identity drift exceeds tolerance:** train/stack dedicated per-character LoRAs on Qwen, or add ControlNet/pose guidance for hard poses.
- **If Wan smooths out the stop-motion look too much:** raise the style-LoRA weight, reduce effective frame rate / add holds, and shorten clips.

## Caveats
- Clip/still counts are arithmetic from the 5s/81-frame native limit; the closest hard practitioner number is "good results up to ~1 minute, color shift beyond." Your actual counts depend on final scene lengths and how aggressively you re-anchor.
- FLF2V "reduces drift / no drift" claims are partly from vendor pages; treat the endpoint-pinning mechanism as solid and marketing superlatives as optimistic. FLF2V still accumulates some drift when many clips are chained — pre-approved keyframes are what keep it in check.
- Several FLUX.2/Qwen VRAM and speed figures come from SEO/vendor blogs; the core specs (parameter counts, licenses, node names, native limits) are corroborated by official ComfyUI, Hugging Face, and Black Forest Labs sources.
- The Reddit-level practitioner detail on chaining was sourced from GitHub issues, Civitai, Hugging Face discussions, and tutorial writeups (Reddit was not retrievable this session), but the failure modes are firsthand user reports, not secondhand summaries.
