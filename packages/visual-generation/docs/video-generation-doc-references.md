# Documentation Reference List: Wan 2.2 FLF2V + Qwen-Image-Edit 2511 Video Pipeline on ComfyUI/RunPod

**TL;DR**
- All seven layers have authoritative, mostly-official documentation; the highest-value primary sources are `docs.comfy.org` (ComfyUI API + Wan/Qwen tutorials), the ComfyUI GitHub source files (`server.py`, `comfy_extras/nodes_video.py`, `comfy_extras/nodes_wan.py`), the Hugging Face model cards, and the official library docs (httpx, Pydantic, Qdrant, Voyage/MongoDB, RunPod).
- The single biggest documentation gap is the exact JSON shape of history `outputs` records for video nodes: the modern built-in `SaveVideo` uses ComfyUI's V3 node schema and returns a `PreviewVideo` UI payload (**not** the legacy `"videos"/"gifs"/"images"` dict), so the developer MUST read source (`comfy_extras/nodes_video.py` and `comfy_api/latest/_ui.py`) to know the exact serialized key — the widely-cited `"gifs"` key belongs to the third-party VHS `VHS_VideoCombine` node, not core SaveVideo.
- Prefer the official ComfyUI native FLF2V path (docs.comfy.org + `nodes_wan.py`) over the kijai wrapper unless a cutting-edge feature is needed; for multi-reference Qwen conditioning, official docs are thin and the best practical references are the lrzjason QwenEditUtils repo and RunComfy node docs.

## Key Findings
- **ComfyUI's native API is small and stable but under-documented.** Five endpoints do the core work (`/prompt`, `/history/{id}`, `/view`, `/upload/image`, `/ws`); the definitive spec is the source (`server.py`), not any doc page.
- **The native SaveVideo output shape is a real trap.** Direct source read confirms native `SaveVideo` returns `io.NodeOutput(ui=ui.PreviewVideo([ui.SavedResult(file, subfolder, io.FolderType.output)]))` — parsing logic that assumes a `"gifs"` or `"videos"` array will silently break unless the workflow uses the VHS node.
- **Wan 2.2 native FLF2V is fully documented and preferred; the kijai wrapper's FLF2V docs are still unfinished.**
- **Multi-reference Qwen-Image-Edit conditioning (character + outfit) has no official node-level doc** — the lrzjason repo is the best resource.
- **Voyage embeds video only on the preview `voyage-multimodal-3.5` model**; stable `voyage-multimodal-3` is text + images only.

## Details

### Layer 1 — ComfyUI native HTTP API
1. **ComfyUI Server API — Routes** — https://docs.comfy.org/development/comfyui-server/comms_routes — Official page stating that `server.py` defines all routes via `@routes` decorators; documents `/ws`, `/prompt` (validation → queue → `prompt_id`), and the execution/queue model. *When: Phase 1, building the async client wrapper for POST /prompt and WebSocket progress.*
2. **`server.py` (repo root)** — https://github.com/comfyanonymous/ComfyUI/blob/master/server.py — Authoritative source defining every route: `@routes.post("/prompt")`, `@routes.get("/history/{prompt_id}")`, `@routes.get("/history")`, `@routes.get("/view")`, `@routes.post("/upload/image")`, `@routes.post("/upload/mask")`, `@routes.get("/object_info")`, `@routes.get("/object_info/{node_class}")`, `@routes.get("/ws")`, `/queue`, `/interrupt`, `/system_stats`. Read for exact query params, multipart fields, dedup/collision behavior, and the binary WebSocket preview header (8-byte prefix: 4 bytes event type + 4 bytes format). *When: Whenever a param or response field is undocumented elsewhere — ground truth.*
3. **Export Workflow (API format)** — documented on DeepWiki "API and Programmatic Usage" (https://deepwiki.com/Comfy-Org/ComfyUI/7-api-and-programmatic-usage) and confirmed on docs.comfy.org — Exported via File → Export (API Format); each key is a node ID, each value is `{class_type, inputs}`, outputs referenced as `[node_id, output_index]`. The API accepts this "API format," NOT the UI-format workflow JSON. *When: Phase 1, templating the workflow JSON you POST.*
4. **REST API Reference (DeepWiki, source-cited)** — https://deepwiki.com/Comfy-Org/ComfyUI/7.1-rest-api-reference — Third-party but tightly source-referenced (cites `server.py` line ranges); documents the newer Jobs API (`/api/jobs`) normalizing queue + history into unified status constants (pending/in_progress/completed/failed/cancelled) via `comfy_execution/jobs.py`. *When: If you want richer status than raw /history polling.*
5. **Workflow JSON Format (DeepWiki)** — https://deepwiki.com/Comfy-Org/ComfyUI/7.3-workflow-json-format — Distinguishes the stripped execution/prompt JSON from the UI workflow JSON; history `outputs` is keyed by node ID. *When: Phase 1–2, parsing history results.*
6. **Subgraph node-ID namespacing** — https://docs.comfy.org/custom-nodes/js/subgraphs — Official: ComfyUI uses three node-identifier types; the compound "locator ID" is `` `${graphId}:${node.id}` `` (this is the `"129:98"` form). Subgraphs each have a UUID and serialize under `definitions.subgraphs[uuid]`. *When: If your exported workflow contains subgraphs and you must address nodes by compound ID.*
7. **Subgraph ID caveat (known limitation)** — https://github.com/Comfy-Org/ComfyUI_frontend/issues/8137 — Documents that the backend uses a compound `subgraph_ID:node_ID` while nodes only know their local `node_ID`, and IDs can collide across subgraphs. *When: Debugging why a subgraph node's history output key doesn't match your expected ID.*

*Community/practical (use only if official is insufficient):* ViewComfy production API guide (https://www.viewcomfy.com/blog/building-a-production-ready-comfyui-api) and 9elements (https://9elements.com/blog/hosting-a-comfyui-workflow-via-api/) — both give complete working Python for `queue_prompt`/`get_history`/`get_image`/`upload_image` and the `history[prompt_id]["outputs"][node_id]` iteration pattern with `if "images" in node_output`.

### Layer 2 — Wan 2.2 FLF2V in ComfyUI
1. **Wan2.2 FLF2V native support (official blog)** — https://blog.comfy.org/p/wan22-flf2v-comfyui-native-support — Announcement of native Wan2.2 14B FLF2V; links to template workflow. *When: Phase 2, orienting on the supported native path.*
2. **Wan 2.1 FLF2V native tutorial** — https://docs.comfy.org/tutorials/video/wan/wan-flf — Official docs; documents `WanFirstLastFrameToVideo` node fusing CLIPTextEncode + CLIPVisionEncode(start/end) + UNETLoader, plus ModelSamplingSD3, CreateVideo, SaveVideo; lists model files. *When: Phase 2, wiring the FLF2V node graph.*
3. **Wan 2.2 14B FLF2V workflow page** — https://comfy.org/workflows/video_wan2_2_14B_flf2v-7016f027bcf1/ — Official workflow; confirms UNETLoader + CLIPLoader (UMT5 XXL FP8 scaled) + VAE + KSamplerAdvanced (SD3 schedule), optional lightning 4-step LoRA. *When: Phase 2, exact node/param reference.*
4. **`comfy_extras/nodes_wan.py`** — https://github.com/comfyanonymous/ComfyUI/blob/master/comfy_extras/nodes_wan.py — Authoritative source defining `WanImageToVideo`, `WanFirstLastFrameToVideo`, and siblings (WanFunControlToVideo, WanCameraImageToVideo, WanVaceToVideo, WanAnimateToVideo, etc.). Read for exact input names/types and the V3 `execute()` signature. *When: Phase 2, when the tutorial omits an input's exact name/type.*
5. **Official Wan2.2 repo** — https://github.com/Wan-Video/Wan2.2 — Alibaba's model repo (Apache 2.0). Per the README, the A14B series uses a two-expert MoE design: "a high-noise expert for the early stages, focusing on overall layout; and a low-noise expert for the later stages, refining video details. Each expert model has about 14B parameters, resulting in a total of 27B parameters but only 14B active parameters per step," with the transition point "determined by the signal-to-noise ratio (SNR)." *When: Background on model architecture and licensing; explains why the workflow loads two UNETs.*
6. **Model files (ComfyUI Wiki, comprehensive)** — https://comfyui-wiki.com/en/tutorial/advanced/video/wan2.2/wan2-2 — Community-maintained but the most complete single reference covering native + Kijai + GGUF + Lightx2v variants and exact file layout (diffusion_models high/low-noise 14B, `umt5_xxl_fp8_e4m3fn_scaled.safetensors` text encoder, `wan_2.1_vae.safetensors`, lightx2v 4-step LoRAs). *When: Phase 2, downloading/placing model files on the network volume.*
7. **kijai/ComfyUI-WanVideoWrapper** — https://github.com/kijai/ComfyUI-WanVideoWrapper — Alternative Wan-only implementation on the frontline of optimizations; use only if you need features not yet in native. Companion weights: https://huggingface.co/Kijai/WanVideo_comfy_fp8_scaled. **Note:** the official docs' WanVideoWrapper FLF2V section is still marked "being prepared," so this path is less documented than native — prefer native, and if you must use the wrapper, read its `example_workflows/` directory directly.

### Layer 6 — Video post-processing (secondary)
1. **ffmpeg official docs** — https://ffmpeg.org/ffmpeg.html and the seeking wiki https://trac.ffmpeg.org/wiki/Seeking — Authoritative for `-ss` (place before `-i` for fast keyframe seek; after `-i` for frame-accurate), `-frames:v 1`, `-q:v`, and the scale filter. Middle-frame thumbnail: `ffmpeg -ss <t> -i in.mp4 -frames:v 1 -q:v 2 thumb.jpg`. *When: Phase 5, extracting a middle-frame thumbnail from the mp4.*
2. **ffmpeg concat** — https://trac.ffmpeg.org/wiki/Concatenate — Official concat demuxer/filter for assembling clips. *When: Phase 5, joining multiple FLF2V clips.*
3. **FILM / RIFE ComfyUI node** — https://github.com/Fannovel16/ComfyUI-Frame-Interpolation — Canonical VFI node set (FILM, RIFE, etc.), category `ComfyUI-Frame-Interpolation/VFI`, needs ≥2 frames. FILM paper: Reda, Kontkanen, Tabellion, Sun, Pantofaru & Curless (Google Research / Univ. of Washington), "FILM: Frame Interpolation for Large Motion," ECCV 2022 (arXiv:2202.04901) — "a frame interpolation algorithm that synthesizes multiple intermediate frames from two input images with large in-between motion... a single unified network... optimize our network with the Gram matrix loss." Google repo: https://github.com/google-research/frame-interpolation. *When: Phase 5, if interpolating/smoothing generated frames.*
4. **Built-in CreateVideo/GetVideoComponents** — defined in `comfy_extras/nodes_video.py` (see Layer 7) — `GetVideoComponents` extracts frames/audio/fps inside ComfyUI, avoiding an external ffmpeg step. *When: Phase 5, if you prefer in-graph frame extraction.*

### Layer 7 — Supporting model docs + the SaveVideo output-shape source
1. **Voyage multimodal embeddings (official reference)** — https://docs.voyageai.com/docs/multimodal-embeddings and API https://docs.voyageai.com/reference/multimodal-embeddings-api — `voyage-multimodal-3` accepts interleaved text + images; `voyageai.Client.multimodal_embed(inputs, model, input_type)`. Token accounting: "every 560 pixels of an image and every 1120 pixels of a video being counted as a token, each input in the list must not exceed 32,000 tokens, and the total number of tokens across all inputs must not exceed 320,000" (images max 16M pixels / 20MB). *When: Phase 6, embedding generation records.*
2. **Voyage multimodal — video support** — https://www.mongodb.com/docs/voyageai/models/multimodal-embeddings/ — The Voyage docs state that **"Video inputs are only supported by `voyage-multimodal-3.5`"** (`Video.from_path` requires `model="voyage-multimodal-3.5"`); stable `voyage-multimodal-3` handles text + images only. `voyage-multimodal-3.5` was announced 15 Jan 2026 (https://blog.voyageai.com/2026/01/15/voyage-multimodal-3-5) and supports 2048/1024/512/256 output dimensions. **Decision point:** to embed the actual video record you must adopt `voyage-multimodal-3.5`; otherwise embed the extracted middle-frame thumbnail with `voyage-multimodal-3`. *When: Phase 6, deciding whether to embed the mp4 directly or a thumbnail.*
3. **Qdrant Python client** — https://python-client.qdrant.tech/ (quickstart https://python-client.qdrant.tech/quickstart.html; module ref https://python-client.qdrant.tech/qdrant_client.qdrant_client.html) — Official; sync + async clients, `query_points`, `create_collection(VectorParams(size, distance))`, `:memory:`/local mode for tests. *When: Phase 6, storing/searching generation-record vectors.*
4. **`comfy_extras/nodes_video.py` (SaveVideo/CreateVideo output shape — DEFINITIVE)** — https://github.com/comfyanonymous/ComfyUI/blob/master/comfy_extras/nodes_video.py — Confirmed by direct read: the built-in `SaveVideo` (V3 `io.ComfyNode` schema) `execute()` returns `io.NodeOutput(ui=ui.PreviewVideo([ui.SavedResult(file, subfolder, io.FolderType.output)]))`. `SaveWEBM` does the same; `CreateVideo` returns a `VIDEO` type via `VideoFromComponents`. So the history `outputs` record for native SaveVideo is produced by `ui.PreviewVideo`, NOT a hand-written `{"videos": [...]}` / `{"gifs": [...]}` dict. *When: Phase 1/2, parsing the video result out of /history.*
5. **`comfy_api/latest/_ui.py` (PreviewVideo/SavedResult serialization — READ THIS)** — in the same repo under `comfy_api/latest/` (imported in nodes_video.py as `from comfy_api.latest import ... ui ...`) — defines how `PreviewVideo` and `SavedResult` serialize into the `ui` dict that lands in history `outputs`. This is the exact file to open to confirm the serialized key name. *When: Phase 1, before hard-coding the history-parsing key for video outputs.*

## Recommendations
1. **Phase 1 (API client): build against `server.py` + docs.comfy.org/comms_routes, and pin your history-parsing to actual observed JSON.** Before writing the video-result parser, run one FLF2V job and dump `history[prompt_id]["outputs"]` verbatim; then confirm the key against `comfy_api/latest/_ui.py`. Do not code to `"gifs"`/`"videos"` from memory.
2. **Phase 2 (Wan): use the native FLF2V template + `nodes_wan.py`.** Only fall back to the kijai wrapper if you hit a missing native feature; if you do, read its `example_workflows/` since docs are unfinished.
3. **Phase 3 (Qwen): use the official native workflow for single-image edits and the lrzjason QwenEditUtils repo for multi-reference (character + outfit) stills.** RunComfy's node docs fill the TextEncodeQwenImageEditPlus gap.
4. **Phase 4 (RunPod): script lifecycle with `runpodctl`, keep all model weights on a network volume, and account for the `/workspace` (Pod) vs `/runpod-volume` (serverless) mount difference.** Remember network volumes lock a pod to one datacenter.
5. **Phase 6 (embeddings): decide the video-vs-thumbnail question now.** If embedding the mp4 itself is a requirement, adopt `voyage-multimodal-3.5`; otherwise embed the ffmpeg-extracted middle frame with the stable `voyage-multimodal-3`.

**Thresholds that change these recommendations:**
- If your exported workflow's save node `class_type` is `VHS_VideoCombine` (common in community Wan graphs), the history key IS `"gifs"` and you should code to that — inspect the API JSON's `class_type` to decide.
- If the native FLF2V template lacks a feature you need (e.g., long-video context windows, specific LoRA stacking), switch to the kijai wrapper path.
- If per-record video semantic search proves necessary (not just thumbnail similarity), migrate the embedding model to `voyage-multimodal-3.5`.

## Caveats
- **Exact native SaveVideo history key is not officially documented.** Source read confirms the `ui.PreviewVideo([ui.SavedResult(...)])` structure, but the final serialized dict key name lives in `comfy_api/latest/_ui.py` — open it before relying on any key. The `"gifs"` key is specifically the third-party VHS `VHS_VideoCombine` behavior (`Kosinkadink/ComfyUI-VideoHelperSuite/videohelpersuite/nodes.py`), not core SaveVideo.
- **Subgraph compound node IDs (`"uuid:node_id"`) in history are not cleanly documented.** See ComfyUI_frontend issue #8137; read `execution.py` for how execution addresses subgraph nodes if your workflow uses subgraphs.
- **WanVideoWrapper FLF2V docs are incomplete** — the official section is a placeholder; treat the wrapper repo's example workflows as the reference.
- **Voyage video embedding depends on a recently-released model** (`voyage-multimodal-3.5`, Jan 2026); verify current availability/stability against docs.voyageai.com before building on it.
- **DeepWiki pages are third-party** (AI-generated over the source). They are useful because they cite exact `server.py`/`execution.py` line ranges, but treat the underlying source files as canonical when they disagree.
- **Several node-level pages (RunComfy) are AI-generated**; they are the best available node references where official docs have no standalone page, but cross-check parameter names against the source `.py` files when precision matters.
